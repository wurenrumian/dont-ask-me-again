# Session Auto Title Design

## Goal

Add automatic session title generation with minimal changes to the current plugin/server architecture.

The title should be generated by the backend automatically when a session begins interacting, using the first user request in that session as the source. The frontend should only add a setting for choosing which existing model-provider configuration is used for title generation, or disabling title generation entirely.

## Scope

This design covers:

- backend-triggered title generation for new sessions
- session metadata storage for generated titles
- frontend settings for choosing a title-generation model
- session list display updated to prefer the generated title over `session_id`

This design does not cover:

- manual title editing
- title regeneration controls
- persistent database storage
- changing the current chat/session protocol

## Current Constraints

- The current server stores session state in `InMemorySessionStore`
- The frontend already fetches model-provider entries from the backend
- The session list currently only exposes `session_id` and `updated_at`
- Main chat requests already create a new session when `session_id` is absent

The implementation should preserve these patterns and avoid introducing a second frontend orchestration flow for title generation.

## Recommended Approach

Use backend-owned asynchronous title generation:

1. When the first user request starts a new session, the backend creates the session as it does today.
2. The backend records the first user instruction in session history as it does today.
3. If a title-generation model has been configured and the session has not already attempted title generation, the backend starts one best-effort background task to generate a short title from that first request.
4. The main chat response proceeds normally and must not wait for title generation to complete.
5. The frontend session list displays the generated title when present, otherwise falls back to `session_id`.

This keeps title generation out of the frontend request flow, avoids slowing down first-turn chat completion, and reuses the existing model-provider registry.

## Data Model Changes

Extend the in-memory session record with lightweight title metadata:

```python
@dataclass
class SessionRecord:
    session_id: str
    created_at: datetime
    updated_at: datetime
    history: list[dict[str, str]]
    title: str | None = None
    title_generation_state: str = "pending"
```

State meaning:

- `pending`: title generation has not yet been attempted
- `running`: a background title task is in progress
- `done`: generation completed, whether title exists or not

Rules:

- each session should attempt automatic title generation at most once
- failure to generate a title should still transition the state away from `pending`
- title generation must never block or fail the main chat request

## Backend Behavior

### Trigger Point

The trigger should live in the existing chat entrypoints after session creation and after the first user instruction is appended, because that is where the backend already knows:

- whether the session is new
- which session record is active
- what the first request text is

The trigger should run only when:

- the session was newly created for this request
- the configured title model is not empty
- the session title state is `pending`
- the extracted first-user text is non-empty

### Generation Strategy

The backend should use a short internal prompt such as:

- summarize the user's first request into a short session title
- return only the title text
- keep it concise
- avoid quotes and trailing punctuation

The output should be normalized before storage:

- trim whitespace
- collapse repeated whitespace
- optionally cap length
- store `None` if the result is empty after normalization

### Runtime Selection

The selected title model should come from the existing model-provider list by id.

The backend should:

- read the saved title-model id from server config
- resolve it to an existing model-provider entry
- temporarily apply that provider configuration only for the title request

If the id is missing, invalid, or points to a deleted model, title generation should be skipped silently.

### Failure Handling

Failures in title generation are non-fatal:

- log the failure
- mark title generation as completed
- leave `title` empty
- do not emit API errors to the frontend for the title task

## API Contract Changes

### Session list response

Extend session entries to include title metadata:

```json
{
  "session_id": "sess_123",
  "title": "Summarize the Nanobot config flow",
  "updated_at": "2026-04-24T10:00:00Z"
}
```

`title` is nullable.

No new dedicated title-generation endpoint is required.

### Settings/config contract

Add one new setting persisted by the plugin:

- `titleGenerationModelId: string | null`

Frontend semantics:

- `null` or empty string means disabled
- any non-empty value must be an existing model-provider id selected from the current backend list

## Frontend Changes

### Settings UI

Add one small section in settings:

- label: `Title generation model`
- control: dropdown populated from existing model-provider entries
- first option: disabled / do not generate

The frontend should not send extra title-generation requests. It only persists the chosen model id in plugin settings.

### Session list display

Wherever session entries are rendered, display:

- `entry.title` when present and non-empty
- otherwise `entry.session_id`

Session identity and selection should continue to use `session_id`, not title text.

## Testing Strategy

Backend tests should cover:

- new session entry starts with `pending` title state
- first request on a new session schedules at most one title attempt
- configured title model disabled skips generation
- generated title is normalized and stored on the session
- generation failure does not break the main chat response
- session list response includes nullable `title`

Frontend tests should cover:

- settings schema default includes disabled title generation
- title-model dropdown includes a disabled option plus existing models
- session list rendering prefers title and falls back to `session_id`

## Tradeoffs

Chosen approach:

- keeps title generation entirely backend-owned
- avoids adding another frontend request path
- minimizes impact on first-response latency

Known limitation:

- titles remain in-memory only, matching the current session-store behavior
- if the server restarts, titles are lost along with the in-memory sessions

This is acceptable because the feature is being added on top of the existing in-memory session model rather than redesigning persistence.
